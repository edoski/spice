Repository guide

Spice Beginner Codebase Guide

Spice is a modular machine-learning system for blockchain fee-decision research. This guide turns the root README and architecture guide into a navigable map, then links to module-level explanations for the code under src/spice.

Root docs: README.md and ARCHITECTURE.md Module pages: 27

How To Read This Codebase

The core pipeline is linear even though the implementation is modular: acquire blockchain blocks, validate them into a corpus, compute features, compile temporal examples, tensorize datasets, train models, decode predictions, evaluate decisions, and store durable results.

Most packages follow the same seam: human config or payload selects an id, the owning package validates the concrete config, a local spec table dispatches to the implementation, and workflows consume a compiled contract instead of concrete implementation details.

Basic Terms

Corpus means canonical block history for one chain/dataset identity. Study means tuning/search state. Artifact means a trained model plus summaries and evaluations. These are separate storage root kinds because they have different lifecycle rules.

Config describes user intent. Contract describes executable behavior after validation. Semantics describes durable meaning so future code can compare datasets, studies, and artifacts without guessing.

Module Map

Foundation

Spice Package

src/spice

This is the top-level Python package. It defines shared vocabulary used across the whole project and points readers toward the major domains: config, acquisition, corpus, temporal problems, modeling, prediction, evaluation, storage, benchmarks, workflows, and CLI.

Core

src/spice/core

Core holds small primitives that many packages need: strict config model behavior, operator-facing errors, safe file replacement, rendering helpers, reporting, path validation, and generic local-spec helpers.

Configuration

Config

src/spice/config

Config turns human-facing names, YAML fragments, workflow selections, and persisted snapshots into typed runtime config objects. It is where string references become validated objects.

Checked-In Config Specs

src/spice/conf

This package stores the YAML specs that users and workflows name. These files are the checked-in catalog of chains, datasets, providers, models, objectives, surfaces, benchmarks, and tuning spaces.

Operator Interface

CLI

src/spice/cli

The CLI is the operator edge. It owns Typer command registration, command defaults, shared options, output formatting, and conversion of domain errors into readable terminal messages.

CLI Commands

src/spice/cli/commands

This package holds concrete command handlers for workflows, config inspection, storage inspection/deletion, transfers, and benchmark operations.

Workflow Runtime

Workflows

src/spice/workflows

Workflows orchestrate complete tasks: acquire a corpus, train an artifact, tune a study, or evaluate an artifact. They connect configs, storage roots, feature building, temporal compilation, modeling, prediction, and evaluation.

Data Pipeline

Acquisition

src/spice/acquisition

Acquisition fetches raw block information from a source and delivers canonical block rows to corpus assembly. It owns block-range planning, retry-aware pulling, and validation of fetched rows.

RPC Acquisition

src/spice/acquisition/rpc

RPC acquisition is the concrete block source that talks to Ethereum-compatible JSON-RPC endpoints through Web3. It enriches block headers with fee-history and priority-fee information needed by later features.

Corpus

src/spice/corpus

Corpus turns acquired block data into a canonical, validated dataset with history/evaluation splits and metadata. It is the durable raw-data foundation for feature building and modeling.

Learning Inputs

Features

src/spice/features

Features convert canonical block rows into numeric columns a model can use. Feature contracts describe required source columns, produced feature names, and semantic fingerprints.

Temporal

src/spice/temporal

Temporal code defines how row-level feature history becomes supervised examples. It describes context windows, candidate decision windows, capability requirements, and compiled problem stores.

Temporal Compilers

src/spice/temporal/compilers

Temporal compilers turn problem configs into compiled problem contracts. They decide how samples, context windows, candidate offsets, and runtime metadata are materialized.

Execution Policy

src/spice/temporal/execution_policy

Execution policies define what a decoded model decision means in time and how to judge whether that decision was reachable, missed a deadline, or mapped to a realized block outcome.

Input Normalization

src/spice/temporal/input_normalization

Input normalization scales feature values so training is numerically stable while avoiding leakage from future data.

Modeling

Dataset Builders

src/spice/modeling/dataset_builders

Dataset builders turn temporal problem stores and feature tables into tensors and sample selections used by training and inference.

Model Families

src/spice/modeling/families

Model families define neural-network architectures, their configs, tuning spaces, tuned parameter types, and construction logic.

Modeling Runtime

src/spice/modeling

Modeling owns training, inference, tuning execution, artifact persistence helpers, runtime planning, batching, scoring, and the bridge between model outputs and objectives/evaluators.

Prediction And Evaluation

Prediction

src/spice/prediction

Prediction defines what the model is trained to output, how batches and targets are represented, how losses/metrics accumulate, and how raw outputs become decoded decisions.

Prediction Families

src/spice/prediction/families

Prediction families are concrete implementations of prediction contracts. They define target construction, output specs, loss, training-state fitting, metrics, and decoding for one task family.

Minimum Block Fee Multitask Prediction

src/spice/prediction/families/min_block_fee_multitask

This concrete prediction family trains the model to choose the candidate block offset with the minimum fee while also learning an auxiliary minimum log-fee regression target.

Evaluation

src/spice/evaluation

Evaluation scores decoded model decisions against temporal fee outcomes. It owns evaluator configs, evaluator contracts, replay adapters, temporal accounting, metric descriptors, and evaluation summaries.

Objectives

src/spice/objectives

Objectives tell training and tuning which metric to optimize and whether bigger or smaller is better. They can point at validation metrics or evaluator-backed economic metrics.

Persistence

Storage

src/spice/storage

Storage owns durable roots for corpora, studies, and artifacts. It manages layout, ids, root-local SQLite state, payload codecs, lifecycle commits, inspection, selection, deletion, and catalog reindexing hooks.

Storage Catalog

src/spice/storage/catalog

The storage catalog is a searchable index of dataset, study, and artifact roots. It records enough metadata for selectors, listing, relationship queries, and refresh/reindex operations.

Operations

Execution

src/spice/execution

Execution submits already resolved workflows to remote targets, runs remote workflow snapshots, records job provenance, follows logs, and transfers storage roots between local and remote locations.

Benchmarks

src/spice/benchmarks

Benchmarks materialize experiment grids into durable workflow plans, submit them, collect evaluation results, and maintain/export result indexes.

Source Docs Used

README.md and ARCHITECTURE.md are the only root markdown files represented here. ADRs, agent docs, progress notes, archives, and benchmark README files are intentionally excluded.
