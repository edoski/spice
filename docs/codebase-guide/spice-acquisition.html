Acquisition

Acquisition fetches raw block information from a source and delivers canonical block rows to corpus assembly. It owns block-range planning, retry-aware pulling, and validation of fetched rows.

src/spice/acquisition Docs: ARCHITECTURE.md

Where It Fits

It is upstream of corpus. It does not decide model features, temporal examples, or storage ids; it supplies clean source rows for corpus code to persist.

Beginner Mental Model

The acquisition layer is the data collector. It asks for block ranges, handles temporary provider failures, validates that the returned rows match the request, and streams them into a sink.

Main Source Files

errors.py
Defines acquisition-specific failures: transient provider errors, oversized requests, and unsupported sources.
pull.py
Implements adaptive range pulling, retry windows, request sizing, and sink publication.
types.py
Defines timestamp ranges, block ranges, pull plans, runtime snapshots, row sources, and evaluation-range helpers.
__init__.py
Package boundary for acquisition imports.

Input And Output Flow

  • A pull plan defines which blocks or timestamps are needed.
  • The controller chooses a batch request size.
  • A block source fetches rows.
  • Rows are validated and handed to a sink for corpus staging.

Common Confusing Concepts

Transient failure

An RPC provider can fail for a while without the request being wrong. The controller shrinks or retries instead of immediately failing the whole acquisition.

Sink

A sink is the receiver of completed batches. Acquisition does not need to know the final storage layout.
