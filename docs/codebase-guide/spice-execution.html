Spice beginner guide

Execution

Execution submits already resolved workflows to remote targets, runs remote workflow snapshots, records job provenance, follows logs, and transfers storage roots between local and remote locations.

src/spice/execution Docs: ARCHITECTURE.md, IMPLEMENTATIONS.md

Where It Fits

It sits below CLI command convenience and above workflows/storage transfer primitives. It should receive explicit targets and resolved config snapshots.

Beginner Mental Model

Execution is the remote-control layer. The workflow is still the same recipe; execution decides where to run it, how to submit it, and how to move the needed roots.

Main Source Files

models.py
Defines SSH, path, workflow, workflow-set, and execution target config models.
provenance.py
Reads execution-related environment variables and builds job provenance records.
remote_runner.py
Remote entry point that hydrates workflow snapshots and runs the requested workflow.
session.py
Opens execution sessions, loads targets, renders SLURM scripts, submits jobs, and checks completion.
submission.py
Submits resolved workflows and emits submission progress events.
transfer_transaction.py
Transfers dataset/artifact roots with local and remote adapters and cleanup rules.
__init__.py
Execution package boundary.

Input And Output Flow

  • CLI resolves config locally.
  • Execution serializes a resolved workflow snapshot.
  • A target session submits a remote runner job.
  • Remote runner hydrates the snapshot and calls workflows.
  • Transfer transactions move storage roots as needed.

Common Confusing Concepts

Resolved workflow only

Remote jobs do not rerun surface selection. They receive the exact resolved payload.

Target

A named execution environment with SSH paths and workflow submission settings.

Provenance

Metadata that records which job, target, and log path produced an artifact or evaluation.
