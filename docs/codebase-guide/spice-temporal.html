Temporal

Temporal code defines how row-level feature history becomes supervised examples. It describes context windows, candidate decision windows, capability requirements, and compiled problem stores.

src/spice/temporal Docs: ARCHITECTURE.md

Where It Fits

It sits after features and before dataset builders/prediction. Temporal subpackages own problem compilers, execution policies, and input normalization.

Beginner Mental Model

The project asks time-based questions: given what was known before a deadline, which future block offset should be selected? Temporal code turns that question into aligned arrays the model and evaluator can share.

Main Source Files

capability.py
Defines TemporalCapability, a compact description of required context and candidate-window behavior.
contracts.py
Compiles problem configs into CompiledProblemContract objects and exposes problem-store contracts.
problem_store.py
Defines the compiled problem store: context summaries, candidate summaries, row-aligned arrays, and validation.
semantics.py
Defines temporal semantic enums such as baseline row mode.
__init__.py
Temporal package boundary.

Input And Output Flow

  • Feature rows are available for a corpus split.
  • A problem compiler chooses samples, context windows, and candidate actions.
  • The compiled problem store aligns feature rows with target/candidate facts.
  • Dataset builders tensorize that store.

Common Confusing Concepts

Context window

Past rows the model is allowed to inspect.

Candidate window

Future action slots the model may choose from.

Temporal capability

A promise about what history and future candidates a downstream workflow needs.
