FABLE lesson 1 · about 8 minutes

Work, price, and priority

A transaction does work. Gas measures the work. Base and priority fees price each unit of work. Keeping those three ideas separate makes the FABLE feature straightforward.

1. A block is a bounded batch

Users submit transactions. A block producer chooses an ordered batch that fits within the block's gas limit. A transfer may consume 21,000 gas; a complex contract call may consume much more. Gas is not money. It is the work meter.

Transaction
Requests state changes.
Gas used
Measures executed work.
Fee
Pays a price for each gas unit.

2. Every included transaction pays two execution-gas prices

The block has one base fee per gas. Every transaction included in that block faces the same base fee. Each transaction can offer a different priority fee per gas.

actual transaction fee = gas used × (block base fee + transaction priority fee)
One transaction. It uses 100,000 gas. The block base fee is 10 gwei/gas. Its effective priority fee is 2 gwei/gas. The effective gas price is 12 gwei/gas, so the transaction pays 1,200,000 gwei, or 0.0012 native tokens.

A transaction's maximum fee fields are spending caps. They are not necessarily what it pays. The base fee consumes the cap first; the remaining room limits the effective priority fee.

3. Why one block has many priority fees

A block has one base fee but many transactions. Urgent users may offer higher tips; patient users may offer lower tips. A block therefore has a distribution of effective priority fees, not one protocol-owned priority fee.

TransactionGas usedPriority fee
A20,0001 gwei/gas
B60,0002 gwei/gas
C20,0005 gwei/gas

The gas-weighted median is 2 gwei/gas: after ordering by priority fee, the cumulative gas crosses half of the block's 100,000 consumed gas inside transaction B. Weighting by gas represents occupied block capacity; counting transactions would treat A, B, and C as equal-sized.

4. What the proposed FABLE feature says

For each closed block, store one value: effective_priority_fee_per_gas_median. It means “the middle effective priority fee across the gas consumed in this block.” The standard eth_feeHistory(..., [50]) RPC returns this value directly.

At decision origin h, blocks through h are known. Their median priority fees may be model inputs. Future blocks h+1 ... h+K are unknown outcomes and cannot enter the input.

Training input
Past median priority fees through h.
Model target
Future minimum base-fee block, unchanged unless the thesis deliberately redefines it.
Evaluation
Recompute user-cost savings with future realized base plus priority fees.

5. Retraining is part of the proposed experiment

The main question is whether priority-fee history improves prediction. Answer it with paired training runs: the existing feature route as control, and the same route plus the median-priority feature as challenger. Keep model family, windows, seeds, tuning policy, and evaluation origins fixed.

A separate diagnostic can take already-trained predictions and add realized priority fees only to the cost calculation. That diagnostic needs no retraining, but it is not the main feature experiment.
