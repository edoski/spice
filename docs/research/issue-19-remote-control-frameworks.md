# Issue 19 remote-control framework deletion test

Research date: 2026-07-13. Scope: one read-only primary-source pass over mature framework candidates for [Issue 19](https://github.com/edoski/spice/issues/19). No framework was installed or executed. This evidence tests the framework alternative; it does not choose between one reduced Session and split direct functions.

## Result

No candidate covers the whole required seam: target-bound OpenSSH behavior, rsync into a hidden destination stage, exact persisted `WorkflowRequest` hydration, Slurm submission/dependencies/machine state, interruption-safe live log following, and remote module/revision lookup. Each leaves material custom implementation or replaces the bounded thesis workflow with a larger runtime/deployment model.

| Candidate | Earned coverage | Custom concepts still required | Added cost | Deletion test |
|---|---|---|---|---|
| Submitit | Slurm script generation, raw dependency parameter, `sacct --parsable2` state, cancellation, result/error capture | SSH target selection, rsync stage/validation/publication, remote revision/module calls, persisted-request hydration, resumable live follow, target error context | Submitit and `cloudpickle` in the cluster Python environment; a shared Submitit folder; its submitted/result pickle contract | Fails |
| Fabric / Paramiko | SSH command execution; SFTP file get/put; partial `ssh_config` interpretation | Slurm rendering/submission/dependencies/state, rsync tree resume/stage validation, exact request hydration, revision lookup, job-aware follow | A second SSH implementation and authentication/config semantics; Paramiko cryptography stack; rsync still remains beside it | Fails |
| Parsl | App/Future dataflow, Slurm provider, worker management, file-staging interfaces | Existing `WorkflowRequest` and physical-object contracts must be wrapped in Parsl Apps/Files/Futures; direct hidden-stage publication, remote revision, and operator follow remain project work | DataFlowKernel, executors, providers, launchers, workers, staging providers, controller/worker communication | Fails |
| `slurmrestd` | Versioned JSON job submit/query/cancel | SSH, rsync, module/revision lookup, request placement/hydration, log access/follow, object validation/publication | Site daemon, JWT or local authentication, TLS/proxy policy, versioned/generated client; no service is deployed on the measured target | Fails |

Submitit is the nearest scheduler library, but it is not a remote-control module. Its own description is a toolbox for submitting Python functions to Slurm, and its executor requires `srun` locally, a cluster-shared folder, and a remote `python -m submitit.core._submit` runtime. It serializes the callable and arguments with `cloudpickle`, then reads a pickled result. That is a second persisted execution contract beside the exact `WorkflowRequest`, not hydration of that request ([Submitit overview](https://github.com/facebookincubator/submitit#what-is-submitit), [Slurm executor source](https://github.com/facebookincubator/submitit/blob/main/submitit/slurm/slurm.py), [serialization source](https://github.com/facebookincubator/submitit/blob/main/submitit/core/utils.py)). Wrapping Submitit itself in SSH preserves the custom SSH seam; submitting a command that hydrates `WorkflowRequest` still pays for Submitit's parallel pickle lifecycle. Its `stdout()` reads the shared log file as a whole rather than providing a reconnectable cross-SSH follow ([Job source](https://github.com/facebookincubator/submitit/blob/main/submitit/core/core.py)). Submitit also invokes plain `sbatch` and parses English `Submitted batch job ...`, whereas the measured target already exposes the smaller supported `sbatch --parsable` interface ([Submitit Slurm source](https://github.com/facebookincubator/submitit/blob/main/submitit/slurm/slurm.py), [supported-interface audit](remote-execution-supported-interfaces-audit.md#supported-public-interfaces)).

Fabric is a credible SSH convenience layer, not a scheduler or rsync replacement. `Connection.run()` executes a remote shell command, while transfer is SFTP/SCP and `put()` accepts one regular file, not a directory ([Fabric `Connection`](https://docs.fabfile.org/en/stable/api/connection.html), [Fabric transfer](https://docs.fabfile.org/en/stable/api/transfer.html)). Fabric uses Paramiko's parser for `ssh_config`; Paramiko explicitly says connections do not themselves use OpenSSH config, and Fabric documents behavior differences such as `ProxyJump` precedence ([Fabric configuration](https://docs.fabfile.org/en/stable/concepts/configuration.html#loading-and-using-ssh-config-files), [Paramiko configuration](https://docs.paramiko.org/en/stable/api/config.html)). Replacing a few `ssh` subprocess calls would therefore discard native OpenSSH behavior while leaving Slurm and rsync custom. Paramiko alone is the same shallower trade with more connection, host-key, streaming, and error handling exposed to project code.

Parsl is credible only if the project adopts its workflow runtime. Official current Slurm examples assume the Parsl program runs on a login node and launch worker blocks through `HighThroughputExecutor` plus `SlurmProvider` ([Parsl cluster examples](https://parsl.readthedocs.io/en/stable/userguide/configuration/examples.html#anvil-purdue)). Its staging model introduces location-independent `File` objects, `Staging` implementations, and staging tasks/Futures. Its rsync provider runs from workers back to an exposed submit-side host, which is not local-to-login transfer into an owner-controlled hidden stage ([Parsl staging](https://parsl.readthedocs.io/en/stable/userguide/advanced/data.html#rsync)). Parsl 1.2 documented an SSH channel, but that interface accepted explicit host/user/port/key inputs and does not establish full native OpenSSH-config equivalence ([historical SSH channel](https://parsl.readthedocs.io/en/1.2.0/stubs/parsl.channels.SSHChannel.html)). Parsl would add more interface than it hides here.

`slurmrestd` is a supported scheduler interface, not a whole remote-compute interface. It provides synchronous, stateless JSON submission and job control, but requires a deployed daemon and authentication; the official Python route generates a version-specific OpenAPI client. SchedMD also recommends a caching proxy to protect the controller ([Slurm REST details](https://slurm.schedmd.com/rest.html), [`slurmrestd(8)`](https://slurm.schedmd.com/slurmrestd.html)). The measured university target has no active `slurmrestd`, so adoption would require administrator infrastructure outside this issue ([supported-interface audit](remote-execution-supported-interfaces-audit.md#measured-surface)). It would still leave every non-scheduler operation custom.

DRMAA was screened out before detailed comparison. The official Python package is beta, last released in 2018, requires a separate `libdrmaa.so`, and names Grid Engine rather than Slurm as its example compatible cluster. Slurm needs an additional non-SchedMD adapter ([`slurm-drmaa`](https://github.com/natefoo/slurm-drmaa)). Even if deployed, DRMAA covers scheduler operations only ([DRMAA Python package](https://pypi.org/project/drmaa/), [DRMAA interface](https://drmaa-python.readthedocs.io/en/latest/drmaa.html)). It is not a mature supported whole-seam candidate for this work.

## Contract consequences

The deletion result narrows later design without deciding it:

- Submit the one persisted typed `WorkflowRequest`; the remote entrypoint hydrates that same request. Do not add callable/result pickle, framework File/Future, or generated REST model persistence.
- Keep scheduler dependencies as Slurm submission input. A workflow DAG runtime adds no leverage for one already-declared dependency string.
- Use the measured machine-readable Slurm surface directly: `sbatch --parsable`, `squeue --json`, and the narrow text `sacct` fallback required by installed Slurm 22.05.8. A library whose state path differs does not delete target-specific verification.
- No candidate supplies reconnectable cross-SSH live follow. Follow must remain explicit scheduler polling plus bounded log attachment, with interruption detaching rather than cancelling the job.
- Rsync remains transport into an owner-selected hidden stage. Destination validation and immutable publication stay typed owner operations; no framework staging interface earns a seam.
- Target configuration remains ephemeral connection/path/resource input. Persist only execution evidence such as job ID and revision, not host, path, URL, framework state, or credentials.

The framework option is unsupported by the deletion test. The remaining owner choice is the shape of the direct implementation: one small target-bound module or concrete transport/scheduler functions. Current caller locality and the final Issue 15 owner operations, not framework capability, should decide that shape.
